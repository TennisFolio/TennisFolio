# QA Checklist

## Purpose

QA는 개발을 대신하는 단계가 아니라, 개발 중 검증이 끝난 기능을 사용자 흐름 기준으로 마지막 확인하는 단계다.

## When To Run

권장 타이밍:

- 기능 구현 완료 직후
- PR 생성 직전
- 큰 설계 변경 후 전체 흐름을 다시 확인해야 할 때

피해야 할 사용:

- 작은 구현마다 반복 실행
- 테스트를 작성하지 않고 QA로 대체
- 아직 기능 흐름이 닫히지 않은 상태에서의 조기 검증

## Flow Checklist

- [ ] 사용자가 진입 지점에서 기능을 찾을 수 있다
- [ ] 사용자가 주요 행동을 완료할 수 있다
- [ ] 성공 결과가 명확하게 보인다
- [ ] 실패 결과가 명확하게 보인다
- [ ] 실패 후 복구 방법이 있다
- [ ] 새 기능이 기존 핵심 흐름을 깨지 않는다

## Backend Checklist

- [ ] 성공 응답 상태와 body가 의도와 일치한다
- [ ] 실패 응답 상태와 body가 의도와 일치한다
- [ ] 인증, 권한, 토큰 조건이 검증됐다
- [ ] DB 저장, 수정, 삭제 결과가 검증됐다
- [ ] 중복 요청이나 잘못된 입력이 처리된다

## Frontend Checklist

- [ ] 초기 화면 상태가 자연스럽다
- [ ] 로딩 상태가 있다
- [ ] 성공 상태가 있다
- [ ] 실패 상태가 있다
- [ ] 모바일과 데스크톱에서 핵심 흐름이 깨지지 않는다
- [ ] 버튼, 입력, 링크의 비활성화 조건이 적절하다

## Integration Checklist

- [ ] FE 요청과 BE API 계약이 일치한다
- [ ] 환경 변수 또는 설정값이 문서화됐다
- [ ] 로컬 실행 조건이 명확하다
- [ ] 기존 테스트가 회귀 없이 통과한다

## gstack QA

gstack `/qa`는 최종 사용자 흐름 검증에 사용한다.

권장 요청 형식:

```text
/qa
기능 문서: docs/features/<feature-name>.md
검증할 사용자 흐름:
1. ...
2. ...
3. ...
중점 확인:
- 성공 흐름
- 실패 흐름
- 회귀 가능성이 큰 기존 흐름
```

## QA Report Template

```md
# QA Report: <기능명>

## Scope

- 검증한 기능:
- 검증한 환경:
- 검증하지 못한 항목:

## Result

- Overall: PASS / FAIL / PARTIAL

## Findings

| Severity | Area | Finding | Suggested Fix |
|---|---|---|---|
| High |  |  |  |
| Medium |  |  |  |
| Low |  |  |  |

## Verified Checklist

- [ ] 성공 흐름
- [ ] 실패 흐름
- [ ] 권한 / 인증
- [ ] DB 변경
- [ ] FE / BE 연동
- [ ] 회귀 확인
```
